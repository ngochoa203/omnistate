WHAT'S ACTUALLY POSSIBLE (via shell.exec workaround)
════════════════════════════════════════════════════════════════════════════════

Technically possible but lacking type-safe APIs and cross-platform abstraction:
  ✅ shell.exec "brew install foo"      (macOS/Linux)
  ✅ shell.exec "apt-get install foo"   (Linux)
  ✅ shell.exec "winget install foo"    (Windows)
  ✅ shell.exec "systemctl start foo"   (Linux)
  ✅ shell.exec "launchctl start foo"   (macOS)
  ✅ shell.exec "powercfg /a"           (Windows power settings)
  ✅ shell.exec "defaults write ..."    (macOS preferences)
  ✅ shell.exec "gsettings set ..."     (Linux settings)

MISSING FROM THESE WORKAROUNDS:
  ❌ Type-safe parameter marshaling
  ❌ Cross-platform abstraction (single API for all OSes)
  ❌ Error handling and validation
  ❌ Human-friendly intent classification
  ❌ Progress monitoring and retries
  ❌ Verification of operation success
